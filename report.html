Analysis

🚌
Bus Overloading
Exceeds maximum capacity, creating safety risks and discomfort for students.
📍
Poor Student Distribution
Some bus routes are overcrowded while others are barely used.
🕐
Lack of Schedule Visibility
Students arrive late or waste time waiting because they don't know when buses leave.
📋
No Assignment Tracking
No record-keeping for delays and breakdowns. No tracking for student locations or travel times on specific routes.
Key Highlights
  • These issues reduce students' trust in the transportation service
  • They increase the workload for administrators
  • A database-backed system is necessary to modernize and streamline the transportation workflow
👤 Student Management
  • Register and manage student profiles
  • Manage subscriptions to transport lines
  • Track history of line changes
  • Identify students without active subscription
🚌 Bus & Fleet
  • Register buses with capacity, brand, status
  • Assign buses to transport lines for periods
  • Track historical bus assignments
  • Monitor fill rate and usage of each bus
🗺 Route & Stop
  • Define transport lines with start/end points
  • Define stops and their order along each line
  • Associate stops with multiple lines
📅 Schedule & Trip
  • Define schedules per line per day of week
  • Generate trips from schedules and bus assignments
  • Record departure times and delays per trip
⚠️ Incident Tracking
⚠️ Incident Tracking
  • Record incidents linked to trip or bus
  • Log delays with cause, duration, affected line
  • Produce list of delayed trips for review
📊 Reporting
  • Display students enrolled per line
  • Calculate bus fill rate
  • List most loaded lines
  • Show schedules and assignment history
Students
  • Can only subscribe to one line at a time
  • May change lines (history must be preserved)
  • Must have valid active subscription to use transport
  • Uniquely identified by student ID
  • Cannot be assigned to a trip without subscription
Routes & Stops
  • Line must serve at least 2 stops
  • Stops are ordered along a line (sequence matters)
  • A stop can belong to multiple lines
  • Each line has a defined start point and endpoint
  • A line must have at least one schedule
Buses
  • Maximum passenger capacity cannot be exceeded
  • Assigned to one line at a time for a given period
  • Bus assignments must be historized
  • Can be reassigned to different line after period ends
  • Must be available before being assigned
Schedules & Trips
  • A line can have multiple schedules per day
  • A trip = one bus on a line at a scheduled time
  • Delays and incidents recorded per trip
  • Trip valid only if bus is assigned to that line
  • Trip history retained after schedule changes
Actor Role Key Interactions
Administrator Manages the entire system Manages students, buses, lines, schedules, assignments, generates reports and handles incidents and delays
Student End user of transport service Views their line and schedule

03

Normalization (BCNF)

All 10 tables satisfy Boyce-Codd Normal Form (BCNF) — achieved by design through surrogate keys, atomic attributes, and the elimination of all transitive dependencies. Surrogate PKs on subscription_id, assignment_id, and trip_id deliberately handle historization requirements.

CREATE TABLE Student (
    student_id   INT             NOT NULL AUTO_INCREMENT,
    last_name    VARCHAR(50)     NOT NULL,
    first_name   VARCHAR(50)     NOT NULL,
    email        VARCHAR(100)    NOT NULL UNIQUE,
    phone_num    VARCHAR(20),
    address      VARCHAR(200),
    status       ENUM('active', 'inactive') NOT NULL DEFAULT 'active',
    CONSTRAINT pk_student PRIMARY KEY (student_id)
);
table_bus.sql 
CREATE TABLE Bus (
    bus_id          INT             NOT NULL AUTO_INCREMENT,
    max_capacity    INT             NOT NULL CHECK (max_capacity > 0),
    bus_brand       VARCHAR(50),
    year_of_service YEAR,
    bus_status      ENUM('available', 'in_service', 'maintenance', 'out_of_service')
                    NOT NULL DEFAULT 'available',
    CONSTRAINT pk_bus PRIMARY KEY (bus_id)
);
table_trip.sql 
CREATE TABLE Trip (
    trip_id                 INT     NOT NULL AUTO_INCREMENT,
    schedule_id             INT     NOT NULL,
    bus_id                  INT     NOT NULL,
    route_id                INT     NOT NULL,
    trip_date               DATE    NOT NULL,
    actual_departure_time   TIME,
    actual_arrival_time     TIME,
    status                  ENUM('scheduled', 'in_progress', 'completed',
                                  'cancelled', 'delayed') NOT NULL DEFAULT 'scheduled',
    CONSTRAINT pk_trip       PRIMARY KEY (trip_id),
    CONSTRAINT fk_trip_sch   FOREIGN KEY (schedule_id) REFERENCES Schedule(schedule_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_trip_bus   FOREIGN KEY (bus_id) REFERENCES Bus(bus_id)
        ON DELETE RESTRICT ON UPDATE CASCADE,
    CONSTRAINT fk_trip_route FOREIGN KEY (route_id) REFERENCES Route(route_id)
        ON DELETE RESTRICT ON UPDATE CASCADE
);
trg_check_active_subscription
Acts as a firewall — prevents a student from having two active lines at once, satisfying the "one active subscription" business rule. Fires BEFORE INSERT on Subscription.
trg_deactivate_old_subscription
Automates the historization process — when a student moves to a new line, their old subscription is automatically closed and dated. Fires BEFORE INSERT on Subscription.
triggers.sql 
DELIMITER $$

-- Trigger 1: a student can only have one active subscription at a time
CREATE TRIGGER trg_check_active_subscription
BEFORE INSERT ON Subscription
FOR EACH ROW
BEGIN
    DECLARE active_count INT;
    SELECT COUNT(*) INTO active_count
    FROM Subscription
    WHERE student_id = NEW.student_id
      AND status = 'active';
    IF active_count > 0 THEN
        SIGNAL SQLSTATE '45000'
        SET MESSAGE_TEXT = 'Error: A student can only have one active subscription at a time.';
    END IF;
END$$

-- Trigger 2: deactivate the old subscription when a new one is inserted
CREATE TRIGGER trg_deactivate_old_subscription
BEFORE INSERT ON Subscription
FOR EACH ROW
BEGIN
    UPDATE Subscription
    SET    status   = 'expired',
           end_date = CURDATE()
    WHERE student_id = NEW.student_id
      AND status = 'active';
END$$

DELIMITER ;

views.sql 
-- View 1: Students per route
CREATE OR REPLACE VIEW v_students_per_route AS
SELECT
    r.route_id,
    r.route_name,
    COUNT(s.student_id) AS nb_students
FROM Route r
LEFT JOIN Subscription s ON r.route_id = s.route_id AND s.status = 'active'
GROUP BY r.route_id, r.route_name;

-- View 2: Bus fill rate per trip
CREATE OR REPLACE VIEW v_bus_fill_rate AS
SELECT
    t.trip_id,
    t.trip_date,
    b.bus_brand,
    b.max_capacity,
    r.route_name,
    ROUND(
        (SELECT COUNT(*) FROM Subscription sub
         WHERE sub.route_id = t.route_id AND sub.status = 'active')
        * 100.0 / b.max_capacity, 2
    ) AS fill_rate_percent
FROM Trip t
JOIN Bus b   ON t.bus_id   = b.bus_id
JOIN Route r ON t.route_id = r.route_id;

-- View 3: Students without an active subscription
CREATE OR REPLACE VIEW v_students_without_subscription AS
SELECT st.student_id, st.last_name, st.first_name, st.email
FROM Student st
WHERE st.status = 'active'
  AND NOT EXISTS (
      SELECT 1 FROM Subscription s
      WHERE s.student_id = st.student_id AND s.status = 'active'
  );

-- View 4: Delayed trips
CREATE OR REPLACE VIEW v_delayed_trips AS
SELECT
    t.trip_id,
    t.trip_date,
    r.route_name,
    b.bus_brand,
    sc.departure_time        AS planned_departure,
    t.actual_departure_time  AS actual_departure,
    TIMEDIFF(t.actual_departure_time, sc.departure_time) AS delay
FROM Trip t
JOIN Schedule sc ON t.schedule_id = sc.schedule_id
JOIN Route    r  ON t.route_id    = r.route_id
JOIN Bus      b  ON t.bus_id      = b.bus_id
WHERE t.status = 'delayed'
   OR (t.actual_departure_time IS NOT NULL
       AND t.actual_departure_time > sc.departure_time);
